# ADR-003: Data Model — Three Tables, String PKs

**Date:** 2026-04-11  
**Status:** Accepted

## Context

We need to persist users, their booking schedules, and an audit log of every booking attempt. The schema must work with both PostgreSQL (production) and SQLite (tests).

## Decision

Three tables managed by SQLAlchemy + Alembic:

### `users`
| Column | Type | Notes |
|---|---|---|
| `id` | TEXT (UUID string) | PK, generated by Python |
| `eversports_user_id` | TEXT | Unique — sourced from Eversports login response |
| `email` | TEXT | Unique |
| `encrypted_password` | TEXT | AES-256-GCM (AESGCM), key from `ENCRYPTION_KEY` env var |
| `created_at` | TIMESTAMPTZ | Set at insert |

### `booking_jobs`
| Column | Type | Notes |
|---|---|---|
| `id` | TEXT (UUID string) | PK |
| `user_id` | TEXT | FK → users |
| `weekday` | INTEGER | 0 = Monday … 6 = Sunday |
| `target_time` | TIME | Class start time, e.g. `18:00` |
| `facility_id` | TEXT | Eversports facility ID, e.g. `"73041"` |
| `class_name` | TEXT | Exact class name as shown on Eversports, e.g. `"CrossFit"` |
| `days_in_advance` | INTEGER | How many days before the class to book, e.g. `4` |
| `enabled` | BOOLEAN | Default `true`; disabled jobs are skipped by the worker |
| `created_at` | TIMESTAMPTZ | Set at insert |

### `booking_logs`
| Column | Type | Notes |
|---|---|---|
| `id` | TEXT (UUID string) | PK |
| `job_id` | TEXT | FK → booking_jobs |
| `executed_at` | TIMESTAMPTZ | When the worker ran |
| `target_date` | DATE | The date of the class that was booked |
| `status` | TEXT | `success`, `failed`, or `already_booked` |
| `message` | TEXT nullable | Order ID on success, error message on failure |

## Why String PKs instead of native UUID type

SQLAlchemy's `UUID` column type behaves differently across databases. Using `String` PKs populated with `str(uuid.uuid4())` in Python works identically on PostgreSQL and SQLite, which is essential for keeping tests fast and dependency-free.

## Consequences

- Alembic manages all schema changes. Run `alembic upgrade head` before deploying a new backend version.
- The `booking_logs` table is append-only — the worker never updates or deletes rows. This preserves the full history and supports the idempotency check in ADR-004.
- There is no soft-delete on `booking_jobs`. Deleting a job cascades to its logs (`cascade="all, delete-orphan"`).
